# 出行规划 Skill（Outing Planner）v2.0

## 适用场景

用户需要规划一段时间的出行活动，包括：
- 周末半日游/一日游
- 带宠物出门玩
- 朋友聚会行程安排
- 探店+活动组合
- 节日/纪念日出行

## 核心原则

1. **按分类推荐，用户自选**：将候选地点按类型分类（景点/商圈/密室/展览/户外等），每类给出 2-3 个带描述的推荐，让用户自己挑选组合，不预先打包方案。
2. **景点必须有描述**：每个地点要给出简短有趣的介绍，不只是名字和评分。
3. **餐厅推荐必须调用 restaurant-recommender skill**：restaurant-recommender 负责小红书找候选 + 高德查验 + 偏好排序；午餐和晚餐都必须推荐。
4. **路线调用 route-planning skill**：用户选定地点后，调用 route-planning 生成交通衔接。
5. **路线图必须生成**：每次行程规划都必须调用 route-visualization 生成路线图 PNG 并通过飞书发送，不可省略。
6. **天气联动**：雨天自动剔除户外活动，优先室内方案。
7. **宠物友好默认**：根据 USER.md，用户养猫，推荐时默认考虑宠物友好。
8. **输出形式：飞书卡片 + 路线图**：在聊天框中展示结构化卡片，并附路线图图片。

## 输入信息

从用户请求和长期记忆中提取：

- **目的地/城市**：默认深圳（根据 USER.md）
- **时长**：半天 / 全天 / 晚间 / 具体小时数
- **人数**：独自 / 两人 / 多人
- **兴趣标签**：美食、拍照、密室、剧本杀、看展、逛街、户外、爬山等
- **预算**：显式或从用户画像推断
- **日期/时间**：工作日 / 周末 / 具体日期
- **宠物同行**：默认 true（根据 USER.md）
- **交通偏好**：地铁优先 / 打车 / 自驾

缺少目的地时，默认深圳；缺少时长时追问。

## 工作流

### Step 1: 候选发现

通过多源搜索收集候选活动/景点：

1. **小红书攻略搜索**（找灵感）：
   - 使用 `xhs search` 搜索 "深圳XX区周末游玩攻略"、"深圳密室推荐"、"深圳看展" 等
   - xhs cli 已配置好，直接调用
   - 提取景点名、商圈名、活动名作为候选
   - 每次搜索 3-5 条笔记即可，不过度抓取

2. **AMap POI 搜索**（验实体）：
   - 使用 `amap-poi` skill 搜索对应景点/商圈/酒店/餐厅的真实 POI 信息
   - 获取坐标、评分、营业时间、地址、电话等基础数据
   - 类型选择：景点 `scenic`，商场 `shopping`，酒店 `hotel`，餐厅 `restaurant`

3. **本地兜底候选库**（降级方案）：
   - 当外部搜索不可用时，使用 `references/fallback_activities.json` 中的本地候选
   - 兜底数据按类型分类：景点、商圈、密室、展览、户外活动

### Step 2: 实体查验

对所有候选活动进行实体信息查验：

- 准确名称
- 地址和坐标
- 评分
- 营业时间
- 是否需要预约
- 标签（室内/户外、宠物友好、适合拍照等）
- 预计游玩时长
- **景点描述**：一段 2-3 句话的介绍，说明这个地方有什么特色、适合什么场景

使用 `amap-poi` 或等效真实地图服务查验。

### Step 3: 天气查询

在推荐前查询天气：

```bash
node /root/.openclaw/workspace/skills/qweather/scripts/weather_qweather.js all --location 深圳
```

根据天气结果调整推荐：
- 雨天：剔除户外活动，优先室内
- 高温：减少户外停留，增加室内空调场所

### Step 4: 按分类输出候选

**这是核心输出步骤。** 不要打包成方案 A/B/C，而是按类型分组，让用户自己选。

#### 输出结构（飞书卡片格式）：

```
🎯 [时长]出行推荐 | [日期/周末]

━━━━━━━━━━━━━━━━━━━

🏞 景点推荐

1️⃣ [地点名] ⭐4.7 | 💰免费 | ⏱2h
   📍 [区域]
   📝 [2-3句介绍：有什么特色、为什么值得去]
   🚇 交通：[最近地铁站/交通方式]
   🐱 宠物友好 / ⚠️ 需确认

2️⃣ [地点名] ⭐4.5 | 💰免费 | ⏱2h
   ...

━━━━━━━━━━━━━━━━━━━

🛍 商圈推荐

1️⃣ [商圈名] ⭐4.6 | ⏱3h
   📝 [介绍]
   ...

━━━━━━━━━━━━━━━━━━━

🎭 密室 / 展览 / 其他活动
...

━━━━━━━━━━━━━━━━━━━

🍽 用餐推荐
👉 根据你选的地点，我可以用 restaurant-recommender 帮你推荐附近餐厅
   （先选地点，后推荐餐厅，这样更精准）

━━━━━━━━━━━━━━━━━━━

💡 下一步
告诉我你想去哪些（比如"1️⃣3️⃣+海岸城"），我帮你：
- 规划最优路线（调用 route-planning）
- 推荐附近餐厅（调用 restaurant-recommender；内部使用小红书候选 + 高德查验）
- 生成路线图（调用 route-visualization）
```

#### 关键规则：

1. **每类推荐 2-3 个**，不要超过 3 个
2. **每个地点必须有描述**（📝 介绍），不能只有名字评分
3. **用户养猫**：默认标注宠物友好状态
4. **不在这里深度推荐餐厅**：先让用户选地点，选好后必须用 restaurant-recommender 推荐对应位置附近的午餐和晚餐
5. **不在这里调用路线规划**：先让用户选地点，选好后用 route-planning 生成路线

### Step 5: 用户选点后 → 路线 + 餐厅

用户选定地点后，进入细化阶段：

1. **调用 route-planning skill**：
   - 传入用户选定的地点列表
   - 获取推荐游玩顺序、每段交通方式、耗时
   - 生成标准 JSON 输出

2. **调用 restaurant-recommender skill**（午餐 + 晚餐）：
   - 传入行程路线中用餐时段的地点坐标/地址
   - 使用 restaurant-recommender 的完整流程：小红书搜笔记 → 提取餐厅名 → 高德查验详情 → 结合偏好排序
   - **午餐和晚餐都必须推荐**，各返回 1-3 个餐厅
   - 午餐地点在行程前半段，晚餐地点在行程后半段/终点附近
   - 不要只返回 amap-poi 的原始 POI 列表；餐厅推荐必须经过 restaurant-recommender 的排序和解释

3. **调用 route-visualization skill**（**必须，非可选**）：
   - 将 route-planning 输出的标准 JSON 传入 `scripts/visualize_route.py`
   - 生成 SVG → PNG 路线图
   - 通过飞书消息发送路线图图片给用户
   - 如果飞书上传失败，回退到文字路线方案

### Step 6: 输出最终行程（飞书卡片 + 路线图）

用户确认后，输出最终行程卡片：

```
🗓 [日期] 行程安排

📍 路线：[地点A] → [地点B] → [地点C]

━━━━━━━━━━━━━━━━━━━

🕘 09:00 | [地点A名称]
   📝 [简短描述]
   🚶 下一站：步行X分钟 / 地铁X站

🕚 11:30 | 🍽 午餐
   [餐厅名] ⭐4.5 | 💰人均XX
   📝 [推荐理由]

🕐 13:30 | [地点B名称]
   ...

🕖 18:00 | 🍽 晚餐
   [餐厅名] ⭐4.5 | 💰人均XX
   📝 [推荐理由]

━━━━━━━━━━━━━━━━━━━

📊 总时长：X小时 | 交通：地铁+步行
🌤 [天气提示]
🐱 [宠物友好说明]

🗺 [路线图图片 — 必须生成并发送]
```

**⚠️ 路线图生成是必须步骤，不是可选的。** 调用 route-visualization skill 的 `visualize_route.py` 生成 PNG 图片，然后通过飞书消息发送。

## 与其他 Skill 的协作

### → amap-poi（真实地点查验）

当需要查验景点、商圈、酒店、餐厅或周边地点时：
- 调用 `amap-poi` skill 的脚本
- 根据意图选择 category：`restaurant` / `hotel` / `scenic` / `shopping` / `cinema` / `transport`
- 只把高德返回的真实名称、地址、坐标、评分、营业时间作为确定信息
- 不把 mock-server 或硬编码兜底结果当成真实地点信息

### → restaurant-recommender（餐厅推荐）

当用户选定地点需要推荐餐厅时：
- 必须调用 restaurant-recommender skill 的完整工作流
- 传入当前位置（用户选定的地点坐标/地址）、用户偏好（从 USER.md 和记忆提取）、预算、宠物友好
- 使用小红书找候选，提取餐厅名，再用高德查验候选 POI，结合偏好排序
- **午餐和晚餐都必须推荐**，各 1-3 个餐厅
- 输出时给出推荐理由、人均、评分、地址/距离和营业时间；不要只贴高德原始结果

### → amap-poi（酒店/民宿基础查验）

当用户需要住宿时：
- 使用 `amap-poi --category hotel` 查询酒店/民宿基础 POI
- 只能确认名称、地址、坐标、评分等基础信息
- 不得声称已确认实时房价、库存、房型或可订状态

### → route-planning（路线规划）

当用户选定地点需要规划路线时：
- 将用户选定的地点列表传给 route-planning skill
- 获取推荐游玩顺序、每段交通方式、耗时、费用
- 获取标准 JSON 输出（`route_plan.json`），传给 route-visualization

### → route-visualization（路线图）— **必须调用**

**每次行程规划都必须生成路线图并发送给用户。** 这是 Step 6 的必须步骤。

- 将 route-planning 的标准 JSON 输出保存为 `route_plan.json`
- 调用 `visualize_route.py <route_plan.json>` 生成 PNG 图片
- 通过飞书消息发送路线图图片（使用 `message` 工具的 `filePath` 参数）
- 如果飞书上传失败，回退到文字路线方案，告知用户

```bash
python3 /root/.openclaw/workspace/skills/route-visualization/scripts/visualize_route.py /tmp/route_plan.json
```

### → qweather（天气）

在行程编排前查询天气：
- 使用 `node /root/.openclaw/workspace/skills/qweather/scripts/weather_qweather.js`
- 传入城市/区域参数
- 根据天气结果调整户外活动安排

## 飞书卡片输出规范

使用飞书 streaming card（流式卡片）输出，格式参考飞书 Adaptive Card 规范。

卡片内容应包含：
- 标题行（emoji + 主题）
- 分类区块（景点/商圈/活动/用餐）
- 每个地点的序号、名称、评分、描述
- 底部引导语（"告诉我你想去哪些"）

发送方式：通过 `openclaw message send --card` 发送 Adaptive Card JSON，或通过 feishu_chat 工具以飞书消息方式发送结构化文本卡片。

## 降级策略

### 小红书搜索不可用

使用 `references/fallback_activities.json` 中的本地候选活动，进入实体查验环节。

### AMap POI 不可用

无法查验实体信息时，只输出候选活动类型和名称，标注"具体信息需要到店前确认"，不包装为确定推荐。

### 天气查询不可用

不主动剔除户外活动，但在方案中标注"建议出发前确认天气"。

### 没有符合条件的活动

主动调整方向：
"同时满足宠物友好、室内、评分高的选择不多。你看是把范围扩大到 X 公里，还是调整活动类型？"

## 禁止行为

- 禁止未经高德 POI 搜索或等价真实数据源查验直接推荐具体场所
- 禁止编造评分、营业时间、门票价格、景点描述
- 禁止未经确认执行预约、购票、下单
- 禁止一次性推荐超过 3 个同类地点
- 禁止把内部工作流日志输出给用户
- 禁止在方案中使用"我已经帮你完成了"等措辞
- 禁止输出"方案 A/B/C"式的打包方案——必须按分类列出，让用户自选
- 禁止只给名字不给描述的景点推荐
- 禁止只推荐午餐不推荐晚餐——**每次行程必须包含午餐和晚餐推荐**
- 禁止跳过路线图生成——**每次行程规划必须调用 route-visualization 生成并发送路线图**
