# 拟人化消费决策 Skill

## 适用场景

当用户表达“去哪吃”“找酒店/民宿”“周末去哪玩”“想探店”“带宠物出门”“别太贵”“附近有什么推荐”等消费决策需求时，使用本 skill。

本 skill 目标是模拟真实人的消费决策过程：先理解约束，再收集候选，再查验实体信息，最后做匹配推荐。

## 核心原则

1. 不直接根据种草内容推荐。
2. 候选 POI 必须经过地图或实体数据查验。
3. 推荐前必须结合用户画像，包括预算、身份、宠物、交通、距离、口味和偏好。
4. 如果关键信息无法查验，应明确告知不确定性，不假装确认。
5. 如果没有符合条件的结果，应主动询问用户是否调整预算、距离、品类或搜索范围。

## 输入信息

优先从用户当前请求和长期记忆中提取：

- 城市或当前位置
- 消费场景：吃饭、咖啡、酒吧、展览、周末出行等
- 预算上限
- 是否带宠物
- 交通方式偏好
- 距离或通勤限制
- 口味、环境、拍照、安静程度等偏好
- 禁忌项，例如不吃辣、不想排队、不想太吵

如果用户说“别太贵”，且没有显式预算，应结合用户画像估算预算。学生或价格敏感用户默认控制在人均 150 元以内。

## 工作流

### 1. 用户画像与约束对齐

在搜索或推荐前，先整理硬约束和软偏好。

硬约束示例：

- 预算上限
- 宠物友好
- 当前城市或可接受距离
- 营业状态
- 是否需要预约

软偏好示例：

- 氛围感
- 拍照好看
- 安静
- 户外座位
- 适合聊天
- 适合独处

若缺少城市或位置，且无法从上下文推断，应先询问用户。

### 2. 泛内容候选收集

可使用小红书、大众点评、内部知识库或本地候选库生成候选 POI。

小红书检索只需要少量候选，通常 3 条帖子即可，不应过度抓取。

如果外部种草搜索不可用，使用 `scripts/fallback_candidates.json` 中的本地候选作为降级来源。

候选输出必须整理成纯 POI 名称列表，不把帖子原文直接作为推荐依据。

### 3. 实体查验

拿到候选 POI 后，必须调用 `amap-poi` 或等价真实地图 API 查验真实信息。优先使用：

```bash
python3 /root/.openclaw/workspace/skills/amap-poi/scripts/amap_poi.py search --keywords "<关键词>" --category <category> --city 深圳
```

按用户意图选择 category：

- 餐厅/饭店/咖啡/夜宵：`restaurant`
- 酒店/民宿/住宿：`hotel`
- 景点/公园/海边/展馆：`scenic`
- 商场/购物中心：`shopping`
- 电影院/娱乐：`cinema`
- 地铁站/火车站/交通枢纽：`transport`

每个候选至少查验：

- 准确名称
- 人均消费或可用的价格字段
- 评分或评价质量
- 地址与距离
- 营业状态
- 标签，例如户外、宠物友好、需预约
- 图片 URL，如 API 支持

不得只凭小红书、点评帖子或模型知识直接推荐。

### 4. 匹配评估

使用 `scripts/rank_pois.py` 对查验后的 POI 做评分和过滤。

过滤优先级：

1. 明显超预算
2. 不满足宠物友好等硬约束
3. 评分过低或评价风险明显
4. 距离过远或交通成本过高
5. 营业状态不确定

推荐时优先给 1-3 个结果，不要堆砌过多选项。

### 5. 输出格式

最终回复应包含：

- 首推结果
- 人均消费
- 评分或口碑
- 位置/距离
- 推荐理由
- 不推荐或被淘汰项的简短原因
- 图片 URL，如可用
- 必要时询问是否需要路线规划、预约建议或替代方案

语气应自然、像真实朋友给建议，不输出内部工作流日志。

## 降级策略

### 地图 API 不可用

如果无法查验地址、坐标、评分/价格字段或营业状态，应停止最终推荐，并说明：

“目前地图数据暂时不可用，我没法确认这些地点的真实地址、营业状态或价格。为避免你跑空，建议稍后再查，或我可以先给你一份待确认候选。”

### 种草搜索不可用

如果小红书或大众点评不可用，使用本地候选库生成候选，再进入地图查验。

### 没有符合条件的 POI

如果所有候选都超预算、不宠物友好或距离过远，应主动询问：

“附近符合宠物友好且人均不高的选择比较少。你想扩大搜索范围、提高一点预算，还是改成外卖/咖啡轻食方向？”

## 禁止行为

- 禁止把 ReAct、Thought、Observation 等内部推理日志输出给用户。
- 禁止未经地图查验直接推荐实体店。
- 禁止编造价格、评分、地址、图片或营业状态。
- 禁止在 API 失败时假装已经查验。
- 禁止一次性推荐过多 POI。
