01 · 引言：为什么我们不做 Demo

过去一年，AI Agent 的 Demo 遍地都是。一个输入框、几个工具、一段漂亮的 System Prompt，就能给老板看出"数字员工"的感觉。
但进入生产，麻烦就来了——Agent 为什么反复调同一个工具？为什么一个任务烧几十万 Token？为什么失败了整条链路都挂了？

这是 Demo 和生产之间的鸿沟。跨过这条鸿沟靠的不是更强的模型，而是背后那个常被忽略的运行时底座——Multi-Agent Harness。

小深（开发代号"小龙虾"）是我们在黑客松 36 小时内构建的一个本地生活管家 Agent，部署在飞书上，能处理餐厅排队、打车、天气、路线规划等复合本地生活场景。

我们的重点不是做一个好看的 Demo，而是把这套东西做成可运行的。下面是我们怎么想、怎么做的。

03 · 记忆体系：三层记忆

Multi-Agent 记忆最常见的失败方式：记得太少（每次都像第一次）、记得太多（上下文膨胀）、不分层（临时和长期混在一起）。

┌─────────────────────────────────────────────────────────┐
│                    Miya 记忆体系                         │
├─────────────────┬───────────────────────────────────────┤
│  Layer 1        │  users/{userId}/session_cache.md       │
│  Session Cache  │  跨轮短期记忆，每轮覆盖写              │
│  （最热）       │  内容：上一轮问题 + 关键信息 + 待跟进  │
├─────────────────┼───────────────────────────────────────┤
│  Layer 2        │  users/{userId}/memory/YYYY-MM-DD.md   │
│  Daily Memory   │  当天事件流水，append-only              │
│  （温）        │  内容：决策、偏好更新、任务结果         │
├─────────────────┼───────────────────────────────────────┤
│  Layer 3        │  users/{userId}/profile.json            │
│  Long-term      │  结构化用户画像，smart_memory.js 更新   │
│  Profile（冷）  │  内容：口味、预算、宠物、交通偏好       │
└─────────────────┴───────────────────────────────────────┘

强制执行机制

记忆读写不依赖 Agent 自觉，而是写进 SOUL.md 的"强制规则"（第 10、11 节），AGENTS.md 的会话启动 checklist 会强制 Agent 在开口回复前读取所有层：

# AGENTS.md 启动顺序（每次对话必须执行）
1. Read SOUL.md          ← 人格
2. Read USER.md          ← 用户基础画像
3. Read ERRORS.md        ← 已知错误禁止行为
4. Read memory/今天.md   ← 当天事件
5. Read session_cache.md ← 上轮摘要

smart_memory.js：自动偏好提取

对话结束后，Agent 调用 smart_memory.js 把本轮对话中用户透露的偏好（如"不吃香菜"、"预算 150"）自动提炼成结构化标签写入 profile.json，无需人工维护。

04 · Skill Registry

Skill 不是普通函数，而是带元信息的可发现能力单元。OpenClaw 在每次会话启动时扫描 workspace/skills/*/SKILL.md，把 frontmatter 注入 Agent 上下文，Agent 按 description 和 read_when 字段自动决定何时激活哪个 Skill。

05 · 任务编排：从意图到执行

我们在 SOUL.md 中定义了 Miya 的四步决策框架，确保每次任务不是随机工具调用，而是结构化的意图→规划→执行→反馈流程：

1. 意图解析
识别场景 + 缺失参数
→
2. 候选发现
xhs / 高德 / 本地库
→
3. 实体查验
POI / 营业 / 评分
→
4. 决策匹配
画像 × 实体 → 推荐

餐厅排队全流程编排（典型场景）

用户："我想去海底捞，人少了叫车提醒我"
    │
    ├─ 意图识别：餐厅监控 + 到号叫车
    ├─ restaurant-queue skill 激活
    │
    ├─ Step 1: take_queue_number
    │   POST /restaurant/海底捞福田/queue/take
    │   → queue_number: "B004", position: 12, estimated_min: 60
    │   → Agent 回复："已取 B 号，当前第 12 位，预计 1 小时"
    │
    ├─ Step 2: 注册 cron 监控（立即执行，不等用户再问）
    │   restaurant_monitor.js register --threshold 5 --queue-number B004
    │   → cron job: queue-monitor-B004，每 30 秒轮询
    │
    ├─ Step 3: monitor_poller（后台，每 30 秒）
    │   GET /restaurant/海底捞福田/queue/B004/status
    │   → position 逐渐减少，随机事件注入
    │
    └─ Step 4: position ≤ 5 时
        → feishu_chat("快到你了！还有约 15 分钟，帮你叫车？")
        → 用户确认后：amap-taxi skill 叫车

出行规划：Skill 编排 Skill（核心亮点）

outing-planner 是一个编排型 Skill——它本身不直接调 API，而是协调多个底层 Skill 完成一次完整的一日游规划。这正是 Multi-Agent Harness "组合能力" 的体现：

用户："周六想带猫去南山玩一天，景点 + 吃饭都安排上"
    │
    ├─ outing-planner 激活（编排器）
    │
    ├─ Step 1: 候选发现
    │   xhs 搜攻略 + 按分类推荐（景点/商圈/展览/密室/餐饮）
    │   → 用户自选组合
    │
    ├─ Step 2: amap-poi 查验每个地点
    │   高德 Place API → 真实坐标/营业时间/评分/是否宠物友好
    │
    ├─ Step 3: route_planning 生成多目的地路线
    │   高德路径规划 API → 各段距离/耗时/交通方式
    │   → 输出标准 route_plan 结构
    │
    ├─ Step 4: route_visualization 可视化
    │   route_plan → HTML 路线图（嵌入飞书消息）
    │
    └─ Step 5: restaurant-recommender 补充餐饮
        在路线节点附近推荐餐厅，结合用户画像（嗜辣/墨西哥菜/预算）
    │
    ▼
输出：飞书卡片（景点+餐厅+预算）+ 路线图 HTML
编排设计：outing-planner / route_planning / route_visualization 之间通过标准化的 route_plan 数据结构解耦（compatible_skills + output_schema 字段声明）。一个 Skill 的输出直接作为另一个 Skill 的输入，不需要胶水代码。

强制进度反馈协议

写入 AGENTS.md 顶部的强制规则，解决"Miya 做着做着就消失"的问题：

# ⚠️ 强制规则：调用慢工具前必须先发消息
调用以下任何工具之前，必须先调用 feishu_chat 发文字消息：
- xhs（小红书搜索，8-15秒延迟）
- agent-browser（浏览器自动化）
- node 脚本（高德、天气）
- 任何可能超过 5 秒的操作

正确顺序：
  feishu_chat("好的，正在搜索小红书，稍等～") → xhs search → feishu_chat("找到结果了") → 输出
收到消息后 15 秒内没有任何回复 = 违规

06 · 沙盒引擎：让随机事件可控演示

演示时最怕的是"等一下 API 没反应"——所以我们做了两种沙盒模式：

模式 A：剧本驱动（SandboxEngine）

读取 JSON 剧本文件（haidilao_demo.json 等），按 tick 推进虚拟时间。每 3 秒一个 tick = 5 分钟虚拟时间。Watcher 注册条件满足时自动触发回调。

适合：演示时完全可控，结果可重现。

curl -X POST localhost:5001/admin/start \
  -d '{"scenario":"haidilao_demo","interval":3}'

模式 B：动态模拟（RestaurantSimulator）

泊松过程模拟到店客人，LLM 每 10-15 个 tick 生成一个随机事件（"前面一桌临时取消"），真实还原餐厅排队的随机性。

适合：真实感演示，每次结果不同。

GET /restaurant/海底捞福田/queue
→ {"waiting": 7, "estimated_min": 35,
    "event": "前面一桌提前离场，预计快了"}

TaxiSimulator：真实价格注入

打车模拟器支持从高德真实 API 获取价格后注入，其他流程（司机分配、位置推进、到达通知）继续模拟。状态机：

→

价格预估
→
driver_assigned
司机接单
→

→

07 · 工具治理

xhs-cli 限速包装器

直接调用 xhs-cli 会触发小红书验证码。我们在挂载卷（持久化，容器重建不丢失）里放了一个 bash 包装器，实现跨进程的全局限速：

# /root/.openclaw/bin/xhs（挂载卷，永久）
LOCK_FILE="/tmp/.xhs_last_call"
MIN_INTERVAL=15

if [ -f "$LOCK_FILE" ]; then
    elapsed=$((now - last))
    if [ $elapsed -lt $MIN_INTERVAL ]; then
        sleep $((MIN_INTERVAL - elapsed + RANDOM % 8))  # 15-23秒随机
    fi
fi
/usr/local/bin/xhs-original "$@"
date +%s > "$LOCK_FILE"  # 命令结束后写时间戳

同时在 Python 层（_common.py）加了 8-15 秒随机初始延迟，确保每次 API 请求之间的实际间隔 ≥ 25 秒。

挂载卷持久化策略

核心教训：所有工具二进制（xhs wrapper、uv、agent-browser）必须安装到挂载卷 /home/ubuntu/.openclaw/，而不是容器文件系统。容器每次重建（加内存限制、修改配置），容器内的安装全部丢失。
工具位置挂载卷
xhs wrapper/root/.openclaw/bin/xhs✅
uv（Python 包管理）/root/.openclaw/bin/uv✅
agent-browser/root/.openclaw/npm-global/bin/✅
xiaohongshu-cli/root/.openclaw/uv-tools/bin/xhs✅
xhs cookie/root/.openclaw/.xiaohongshu-cli/cookies.json✅（symlink）

09 · 技术亮点与特色

🧠 三层记忆 + 强制读写

Session Cache（跨轮）、Daily Memory（当天流水）、Long-term Profile（结构化画像）。写入不靠 Agent 自觉，SOUL.md 硬性规则强制执行。

📋 Skill Registry 自动发现

SKILL.md frontmatter 机制，Agent 按 read_when 字段自动激活 Skill，不需要人工维护工具清单。新增 Skill 只需放文件。

🎲 双模式沙盒

剧本驱动（可重现演示）+ 动态模拟（泊松过程 + LLM 随机事件）。打车支持真实价格注入，其余流程模拟。

⏱ 三层兜底降级

xhs-cli（真实 API）→ agent-browser（浏览器抓取）→ web_search（搜索引擎）。任何一层失败自动降级，不向用户暴露技术细节。

🔒 工具限速治理

xhs 跨进程全局限速（文件锁 + 时间戳），Python 层 8-15 秒随机延迟。防验证码触发，保护账号安全。

📡 强制进度反馈

AGENTS.md 顶部强制规则：调用慢工具前必须先发 feishu_chat 消息。15 秒无回复 = 违规。解决"Miya 沉默消失"问题。

💾 挂载卷持久化

所有工具二进制装在 /home/ubuntu/.openclaw/（挂载卷），docker compose recreate 不丢失。docker-compose.yml 配置 PATH 和 UV_TOOL_DIR 指向挂载卷。

📝 ERRORS.md 知识库

已踩过的坑（xhs IP 封禁、验证码处理、不要装在容器文件系统）写入 ERRORS.md，Agent 每次启动必读，不重蹈覆辙。
