# restaurant-recommender Skill · 视频讲解逐字稿

> 约 2.5 分钟。括号里是动作提示，不用念出来。语速放松，像跟朋友讲。

---

**（开场 · 对着镜头或飞书界面）**

大家好。我来讲一下我们 Miya 里我最满意的一个技能——**餐厅推荐**。

先说个痛点：现在找餐厅有多难？小红书翻半天，全是软广和滤镜；大众点评看评分，又看不出哪家适合我自己的口味。信息散在好几个 App 里，最后还是靠自己拍脑袋。

我们这个 `restaurant-recommender` 想解决的就是这件事——**它不是简单地搜出来给你看，而是把三种数据交叉融合之后，给你一个真正适合你的答案。**

**（停顿，引出三个数据源）**

哪三种数据？

第一，**小红书的真实用户口碑**——解决"哪家是真的好吃，哪些是雷"。
第二，**高德地图的客观数据**——评分、人均、营业时间、距离，解决"这家店真实存在、还在营业、价格合不合适"。
第三，**我自己的口味画像**——Miya 记得我嗜辣、爱吃墨西哥菜和小龙虾、预算大概多少。

单看小红书，可能是软广，也可能店早就黄了；单看高德，又没有真实体验。**只有两边交叉验证，再叠上我的偏好，结果才靠谱。** 这正是 Agent 比单个 App 强的地方。

**（切到飞书，现场发一句话）**

我们直接跑一次。我在飞书跟 Miya 说——"帮我找南山区好吃的，预算两百以内"。

**（边等边讲她在做什么）**

大家注意她不是默不作声，而是一步步告诉我进度。她现在在做的是这样一条流水线：

**第一步**，她搜小红书，搜两轮——一轮按热门、一轮按最新，然后过滤出南山、蛇口、后海这些区域相关的笔记。

**第二步**，她逐条读笔记正文，用正则把里面提到的餐厅名提取出来，**还会统计每家被提到几次**——被提到越多，说明越热门，这个会算进后面的排序权重。

**第三步**，把这些餐厅名拿去高德查真实信息：评分、人均消费、营业时间、离我多远。**这一步是关键的"查验"**——确保推给我的店是真实存在、还在营业的，不是小红书上一张过期的照片。

**第四步**，按我的画像打分排序。我嗜辣，所以火锅、川菜、湘菜会加权；爱海鲜也加权；小红书里被反复提到的店再加分。最后排出前几名给我。

**（指向 Miya 的输出结果）**

你看结果——每一家都有名字、真实评分、人均、距离，还附了一句小红书的真实评价摘要。这就是三方数据融合之后的样子。

**（收尾 · 讲两个工程亮点）**

最后讲两个细节，体现我们做的不是一次性 Demo，而是真能跑的东西。

一个是**三层兜底**：万一小红书被风控了，它会自动降级用浏览器抓，再不行就用搜索引擎，绝不会直接报错挂掉。

另一个是**限速保护**：调用小红书有跨进程的文件锁，控制频率，避免触发验证码——这是我们踩过坑之后加的。

**（结尾）**

所以总结一句：**别人的推荐是搜出来给你看，我们的推荐是主观口碑、客观数据、加上你的口味，三方交叉验证出来的。** 这就是 restaurant-recommender。谢谢！

## 录制提示

- 全文约 850 字，正常语速 2.5 分钟，赶时间可砍掉"第二步统计提及次数"那段压到 2 分钟
- **三个反复强调的词**：交叉验证 / 数据融合 / 查验
- 录之前先在飞书发一次确认 Miya 能正常跑（`xhs status` 通），万一当场被风控，提前留个录屏兜底
- 最有说服力的画面：Miya 的进度反馈 + 最终带真实评分人均的结果卡片

## 技能四步流水线（对照讲解）

| 步骤 | 做什么 | 数据源 |
|------|--------|--------|
| Step 1 | 小红书双路搜索（热门 + 最新）+ 区域过滤 | 小红书 xhs-cli |
| Step 2 | 正则提取餐厅名 + 统计提及次数 | 笔记正文 |
| Step 3 | 高德 POI 查验（评分/人均/营业时间/距离）| 高德 Place API |
| Step 4 | 按用户画像打分排序（嗜辣 +15 / 海鲜 +10 / 提及次数 +15）| profile.json |

## 三层兜底降级

```
小红书 xhs-cli（真实 API）
    ↓ 失败（验证码 / IP 风控）
agent-browser（浏览器抓取）
    ↓ 失败
web_search（搜索引擎，保底）
```
